# i18n Conventions

## General

- All user-facing strings must use translation functions — never hardcode text in components
- Languages: English (`en`) and French (`fr`), fallback to `en`
- Two namespaces: `translation` (default, UI) and `email` (email templates)
- Translation files in `app/locales/`
- Keys must be **alphabetically sorted** in translation files
- Add keys to **all language files** when creating translations

## Key Naming

- Flat dot-notation with feature-based prefixes: `<feature>.<section>.<item>`
- Examples:
  - `event-management.proposals.list.items`
  - `auth.signin.heading`
  - `common.confirmation.delete`
  - `settings.preferences.language`
- Use `common.*` for shared/reusable strings

## Usage in Components (Client-side)

- Use `useTranslation()` hook without parameters (default `translation` namespace)
- Access with `t('key')`: `const { t } = useTranslation()`
- Interpolate variables: `t('auth.signin.auth-provider', { label: 'Google' })`
- Pluralize: `t('admin.teams.total', { count: 5 })`
- Format lists: `t('common.by', { names: ['Alice', 'Bob'] })`

## Usage in Server Code (Loaders/Actions)

- Access via middleware context: `const i18n = getI18n(context)` from `~/shared/i18n/i18n.middleware.ts`
- Never create i18n instances directly in server code
- Get locale: `const locale = getLocale(context)`
- Translate for locale: `const t = i18n.getFixedT('fr')`

## Usage in Emails

- Use `getEmailI18n(locale)` from `~/shared/i18n/i18n.emails.ts` for `email` namespace
- Email translations use recipient's `users.locale` from DB
- Use `interpolation: { escapeValue: false }` when translations contain HTML

## Language Detection

- Detection order: (1) `locale` cookie → (2) `Accept-Language` header → (3) fallback `en`
- Server sets `<html lang>`, client reads it
- Language preference stored in `locale` cookie and `users.locale` DB field

## Type Safety

- Translation keys type-checked via `i18next.d.ts` declaration
- English resources define type contract — all languages must match same keys
